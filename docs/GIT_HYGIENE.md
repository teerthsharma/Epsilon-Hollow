# Git Hygiene for Seal OS Contributors

> **Rule of thumb:** If you didn't write it by hand or it can be regenerated by a build command, it probably doesn't belong in git.

---

## 1. The Golden Rules

| ✅ Do | ❌ Don't |
|-------|----------|
| Commit source code, documentation, and tests | Commit build artifacts (`*.exe`, `target/`, `*.rmeta`) |
| Commit `Cargo.lock` at workspace root and excluded crates | Commit member `Cargo.lock` if the member is inside a workspace |
| Commit `.env.example` with dummy values | Commit `.env` with real URLs, keys, or secrets |
| Commit small, focused changes with clear messages | Commit giant blobs (PDFs > 1 MB, ML models, ISOs) |
| Use `git status` before every commit | Assume GitHub Desktop / VS Code handles ignores perfectly |

## 2. Why GitHub Desktop Shows Ignored Files

If you see files like:
```
kernel/seal-os/target-nightly-check-abs/debug/deps/libproc_macro2-...
```
in GitHub Desktop even though `.gitignore` exists, it's usually one of these:

### A. GitHub Desktop Cache Stale
**Fix:**
1. Close GitHub Desktop completely.
2. Re-open it. The cache refreshes on launch.

### B. File Was Tracked Before `.gitignore` Was Added
**Fix:**
```bash
git rm --cached <path-to-file>
git commit -m "chore: remove accidentally tracked build artifact"
```

### C. Pattern Doesn't Match Deep Paths
**Fix:** Ensure `.gitignore` uses `**/` for recursive matching:
```gitignore
# Good — catches target/ at any depth
**/target/

# Good — catches target-nightly-foo/ at specific location
kernel/seal-os/target-*/
```

### D. Multiple `.gitignore` Files Conflict
Search for nested `.gitignore` files that might be overriding the root:
```bash
find . -name ".gitignore" | grep -v node_modules | grep -v target
```

## 3. Pre-Commit Checklist

Run this before every commit:

```bash
# 1. See exactly what will be committed
git status --short

# 2. Verify nothing looks like a build artifact
git diff --cached --stat

# 3. If you see target/, .env, or binaries:
git reset HEAD <file>
# Then add the correct pattern to .gitignore

# 4. Check for large files (> 100 KB)
git diff --cached --stat | awk '{print $3, $1}' | sort -rn | head
```

## 4. Common Seal OS Scenarios

### Build Artifacts

Seal OS uses both **stable** and **nightly** Rust. This creates multiple target directories:

| Directory | Why It Exists | Ignored? |
|-----------|---------------|----------|
| `target/` | Default cargo output | ✅ Yes |
| `target-test-*/` | Custom test targets | ✅ Yes |
| `kernel/seal-os/target-nightly-*/` | Nightly `cargo check` / `cargo test` | ✅ Yes |
| `kernel/seal-os/target/**/vbox-smoke/` | VM smoke test outputs | ✅ Yes |
| `kernel/seal-os/target/**/qemu-proof/` | QEMU proof bundles | ✅ Yes |

If a new `target-*` directory appears, add it to `.gitignore` immediately.

### Environment Files

```bash
# NEVER commit
.env
.env.local
.env.production

# ALWAYS commit (as documentation)
.env.example
.env.sample
```

### Cargo.lock Policy

| Location | Commit? | Reason |
|----------|---------|--------|
| Root `Cargo.lock` | ✅ Yes | Workspace reproducibility |
| `kernel/seal-os/Cargo.lock` | ✅ Yes | Excluded from workspace, needs its own lock |
| `kernel/seal-mkimage/Cargo.lock` | ✅ Yes | Standalone tool |
| `tools/nv-4060-probe/Cargo.lock` | ✅ Yes | Standalone tool |
| Workspace member `Cargo.lock` | ❌ No | Redundant with root lock |

## 5. Large File Hygiene

Git is not designed for large binaries. If you accidentally commit a 4 MB test binary, it stays in history forever (unless you rewrite history, which is dangerous on `main`).

**Prevent it:**
```bash
# Check for large blobs already in history
git rev-list --objects --all \
  | git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' \
  | awk '$1 == "blob" && $3 > 100000 {print $3, $4}' \
  | sort -rn | head -20
```

**If you find a large file that shouldn't be there:**
- If it's on a feature branch: `git filter-repo` or BFG Repo-Cleaner
- If it's on `main`: Document it honestly and don't make it worse. Rewriting `main` breaks everyone's clone.

## 6. Line Endings

Seal OS uses LF line endings (Unix style). The root `.gitattributes` enforces this:

```gitattributes
* text=auto eol=lf
*.bat text eol=crlf
*.cmd text eol=crlf
```

If you develop on Windows:
```bash
# Ensure git doesn't convert LF to CRLF on checkout
git config --global core.autocrlf false

# Or, if you need CRLF for other projects:
git config --local core.autocrlf false
```

## 7. Commit Message Style

We follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <description>

<body>

<footer>
```

Examples:
```
feat(gpu): add RTX 4060 device IDs to NVIDIA scanner

fix(memory): prevent double-free in TopoRAM repair path

docs(readme): update GPU status from "software only" to "infra ready"
```

## 8. Quick Hygiene Commands

```bash
# Remove all untracked files (DANGEROUS — review first!)
git clean -fdxn   # dry run
git clean -fdx    # actually delete

# Remove build artifacts without touching tracked files
find . -type d -name "target*" -not -path "*/.git/*" -exec rm -rf {} +

# Verify .gitignore works for a specific file
git check-ignore -v path/to/file

# See which files are tracked but match .gitignore patterns
git ls-files -i --exclude-standard
```
